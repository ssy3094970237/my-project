## 一、 项目背景与设计目标

随着宠物经济的蓬勃发展，越来越多的宠物主开始追求更加精细化、便捷化的宠物用品选购体验。《猫咪购物园》小程序正是在这种背景下诞生的一款模拟商城产品。
本小程序的视觉风格定位为“自然、温馨、清新”，主色调采用低饱和度的天然绿灰色（`#93a07d`），以期给用户带来如同置身于森林公园般的温和视觉体验。

在功能架构上，项目不仅需要提供静态的商品展示，更需要提供完整的业务链路闭环：从商品浏览、分类切换，到商品详情阅读、购物车动态加减结算，再到地址簿的 CRUD（创建、读取、更新、删除）管理与订单模拟支付。在不依赖真实后端的情况下，纯前端的状态一致性、数据校验以及容错逻辑，成为了保证小程序高质量交付的关键。

## 三、 核心代码实现与架构分析

为了详细展示项目的技术实现细节，以下对购物车结算模块以及收货地址管理模块的核心 JavaScript 代码及业务流向进行深度剖析：

### 1. 购物车结算系统核心代码 (`pages/car/car.js`)

购物车页面的核心逻辑是管理购物车数组 `cartData` 的渲染、移出、全选以及动态总价计算，并在结算时将已购商品组装转换为订单流，写入全局 `globalData.allOrder` 中。

#### (1) 核心状态读取与实时总价计算机制
在页面显示时，需要同步全局的购物车商品数据，并调用核心计算函数 `calcTotal`：

```javascript
// 每次显示页面时，同步全局内存级单一数据源中的商品列表
onShow() {
  var app = getApp();
  this.setData({
    cartData: app.globalData.goodsData || []
  });
  this.calcTotal();
},

// 核心计算函数：遍历购物车列表，仅累加已勾选（checked == true）的商品金额
calcTotal: function() {
  let total = 0;
  let list = this.data.cartData;
  for (let i = 0; i < list.length; i++) {
    if (list[i].checked) {
      // 动态累加：单价 * 数量
      total = total + list[i].num * list[i].price;
    }
  }
  // 更新页面渲染
  this.setData({
    totalMoney: total
  });
}
```

#### (2) 购物车商品的单项删除与内存同步
用户点击“移出”时，会通过 `dataset` 传入索引 `index`，对内存进行裁剪：

```javascript
delete: function(e) {
  let index = e.currentTarget.dataset.index;
  let delegoods = this.data.cartData;
  
  // 裁剪数组
  delegoods.splice(index, 1);
  this.setData({
    cartData: delegoods
  });
  
  // 同步写回全局 globalData，保持多页面数据高度一致
  getApp().globalData.goodsData = delegoods;
  
  // 重新计算总价
  this.calcTotal();
  
  wx.showToast({
    title: '移出成功',
    icon: 'success',
    duration: 2000
  });
}
```

#### (3) 订单生成与模拟结算逻辑
当用户点击“去结算”时，前端逻辑会过滤出被勾选的商品，生成格式化的订单流并追加到全局的订单池 `allOrder` 中，同时清理购物车中的已购项：

```javascript
goPay: function() {
  let selectArr = [];
  let allCart = this.data.cartData;
  
  // 1. 筛选被勾选商品
  for (let i = 0; i < allCart.length; i++) {
    if (allCart[i].checked) {
      selectArr.push(allCart[i]);
    }
  }
  
  // 2. 异常拦截：未勾选任何商品直接结算
  if (selectArr.length == 0) {
    wx.showToast({
      title: "请勾选商品",
      icon: "none"
    });
    return;
  }

  // 3. 组装并生成订单，写入全局订单池
  if (!getApp().globalData.allOrder) {
    getApp().globalData.allOrder = [];
  }
  let orderPool = getApp().globalData.allOrder;
  for (let j = 0; j < selectArr.length; j++) {
    let item = selectArr[j];
    orderPool.push({
      id: Date.now() + j, // 引入时间戳生成模拟订单ID
      shop: item.shop,
      name: item.name,
      desc: item.desc,
      num: item.num,
      price: item.price,
      url: item.image,
      status: 1,         // 状态：1 代表已付款
      statusText: "已付款"
    });
  }
  getApp().globalData.allOrder = orderPool;

  // 4. 清理购物车已勾选的商品，保留未结算商品
  let newCart = [];
  for (let k = 0; k < allCart.length; k++) {
    if (!allCart[k].checked) {
      newCart.push(allCart[k]);
    }
  }
  this.setData({
    cartData: newCart,
    totalMoney: 0,
    isAllCheck: false
  });
  getApp().globalData.goodsData = newCart;

  wx.showToast({
    title: "下单成功",
    icon: "success"
  });
  
  // 延迟1秒跳转，提供优秀的UX操作反馈缓冲
  setTimeout(function() {
    wx.navigateTo({
      url: "/pages/orders/orders"
    });
  }, 1000);
}
```

### 2. 收货地址管理系统核心代码 (`pages/address/addAddress/addAddress.js`)

地址簿的核心逻辑在于收集表单输入、提供容错防御性校验（如手机号格式、必填项），并通过本地持久化缓存（`wx.setStorageSync`）实现数据的断电留存。

```javascript
// 收集表单输入字段值
inputName: function(e) {
  this.setData({ name: e.detail.value });
},
inputPhone: function(e) {
  this.setData({ phone: e.detail.value });
},
inputDetail: function(e) {
  this.setData({ detail: e.detail.value });
},

// 保存收货地址逻辑
saveAddress: function() {
  // 1. 结构化地址数据对象
  let address = {
    id: Date.now(), // 动态注入唯一ID，便于后续精准匹配和编辑删除
    name: this.data.name,
    phone: this.data.phone,
    region: this.data.region,
    detail: this.data.detail
  };
  
  // 2. 防御性健壮校验
  if (!address.name || !address.phone || !address.detail || address.region.length === 0) {
    wx.showToast({
      title: '请填写完整的收货信息',
      icon: 'none'
    });
    return;
  }
  
  // 手机号 11 位正则强验证
  let phoneReg = /^1[3-9]\d{9}$/;
  if (!phoneReg.test(address.phone)) {
    wx.showToast({
      title: '请输入合法的11位手机号',
      icon: 'none'
    });
    return;
  }

  try {
    // 3. 读取本地历史地址列表，若为空则初始化为数组
    let addresses = wx.getStorageSync('addresses') || [];
    
    // 4. 追加新地址并重新存入持久化本地存储中
    addresses.push(address);
    wx.setStorageSync('addresses', addresses);
    
    wx.showToast({
      title: '保存成功',
      icon: 'success',
      duration: 2000
    });
    
    // 5. 返回地址列表页
    wx.navigateTo({
      url: '../../address/address',
    });
  } catch (e) {
    console.error("写入地址本地存储失败:", e);
    wx.showToast({
      title: '系统故障，保存失败',
      icon: 'none'
    });
  }
}
```

## 四、 开发过程中遇到的最大困难及解决过程

在没有传统后端与数据库支撑的情况下，如何仅利用微信小程序的本地特性（如全局变量、本地缓存、生命周期钩子）来实现健壮的、多页面间互通的状态管理，是我遇到的最大技术瓶颈。

### 1. 困难一：购物车状态在多页面切换时的不同步问题
* **问题描述**：
  用户在“商品详情页”点击“加入购物车”后，退回到“商品分类页”继续浏览，再切换到 Tab 页“购物车”时，发现刚才添加的商品没有显示出来；或者在购物车中修改了购买数量，回到商品页再次点击进入同一个详情页时，显示的仍是默认数量。
* **排查与诊断**：
  经过对微信小程序运行机制的分析，我发现 Tab 页面在切换时通常触发的是 `onShow` 生命周期，而不会重新触发 `onLoad`。原来的数据加载逻辑全部写在 `onLoad` 中，导致页面在内存中被复用时没有去读取最新的购物车状态。此外，直接频繁写本地缓存（`wx.setStorageSync`）容易造成高频 I/O 阻塞，导致界面渲染出现肉眼可见的延迟。
* **解决方案**：
  - **引入全局状态（GlobalData）**：在宿主实例 `app.js` 的 `globalData` 中定义一个共享的 `goodsData` 数组作为“内存级单一数据源”。
  - **生命周期钩子优化**：将数据读取和渲染逻辑从 `onLoad` 转移到 `onShow`。每次购物车页面或商品详情页“显示”（`onShow`）时，都主动从 `app.globalData.goodsData` 中获取最新数据并调用 `this.setData` 刷新视图。
  - **持久化策略**：为了防止小程序进程被系统杀死导致购物车数据丢失，我引入了“退出时持久化”机制——仅在组件的 `onUnload`（卸载）或页面的 `onHide`（隐藏）时，才同步将内存中的购物车列表写入本地缓存（Storage），大幅降低了对本地存储的读写频率。

### 2. 困难二：复杂地址簿的动态增删改与多层回调异常
* **问题描述**：
  在地址编辑页面（`editAddress`）中，用户修改了地址后点击保存，返回地址列表页时，列表未能即时高亮展示修改后的地址。有时在删除地址后，由于没有妥善清理关联关系，导致订单确认页依然关联着已被删除的地址，引发了空指针渲染报错。
* **解决方案**：
  - **唯一 ID 绑定机制**：使用时间戳生成唯一 `id` 给每个地址项。在地址修改和删除时，通过 JavaScript 的 `Array.prototype.findIndex` 和 `splice` 方法进行高精度匹配操作。
  - **防御性空指针校验**：在订单页面渲染地址时，增加判断逻辑：
    ```javascript
    const activeAddress = addressList.find(item => item.id === selectedId);
    if (!activeAddress && addressList.length > 0) {
      // 若原选择地址被删除，默认降级选择列表首个地址
      this.setData({ address: addressList[0] });
    } else {
      this.setData({ address: activeAddress || null });
    }
    ```
    通过这类防空校验，彻底消除了由于数据不一致导致的前端页面崩溃风险。

## 五、 对 AI 辅助编程的心得体会与深度思考

本次实训中，我深度依赖了 AI（如 Codex/Cursor/Trae 等）进行结对编程（Pair Programming），这让我对现代软件开发模式产生了全新的认识：

1. **AI 是极佳的效能放大器**：
   微信原生开发中，大量的 WXML 组件属性绑定和 WXSS Flex 布局规则非常琐碎。以往需要花费大量时间查阅微信官方开发文档，而现在只需要通过一句精准的 Prompt，AI 就能在数秒内输出结构严谨的骨架代码。这让我的精力能够从低效的“搬砖”工作中解放出来，专注于整个小程序的路由规划、状态流动设计以及异常边界防御。
2. **AI 输出不能盲信，人机协作需要“批判性思维”**：
   在开发期间，AI 曾经给出过一些看似完美但暗藏缺陷的代码。例如，在购物车计算总价时，AI 给出的示例未考虑浮点数精度丢失问题（如 `0.1 + 0.2 = 0.30000000000000004`），导致购物车结算金额有时会出现一长串小数。我发现该问题后，主动引入了 `.toFixed(2)` 对金额进行规范格式化。这说明开发者必须具备过硬的专业基础，将 AI 生成的代码作为参考，在批判、重构中落地。
3. **AI 辅助的代码审查是极佳的学习途径**：
   在项目后期，我利用 AI 运行了 Code Review。AI 能够以极高的冷酷度指出我代码中的性能隐患，比如高频触发 `setData` 带来的渲染阻塞，以及 Storage 操作未做 `try...catch` 容错。这种即时的反馈让我能够以极快的速度吸收行业内最佳实践，相当于身边随时配有一位资深的架构师进行手把手指导。
